# 架构说明

## 目标

构建一个可复现的政府采购围串标审查 harness。这个 harness 必须让 agent 或人工审查员都能重复运行同一套流程，并查看每一条怀疑结论背后的具体证据。

## 为什么采用这种形态

本仓库借鉴了 OpenAI 强调的两个思路：

- 在 [Harness engineering](https://openai.com/zh-Hans-CN/index/harness-engineering/) 中，仓库本身就是记录系统，配合简短的 `AGENTS.md`、结构化文档和显式执行计划。
- 在 [Unrolling the Codex agent loop](https://openai.com/zh-Hans-CN/index/unrolling-the-codex-agent-loop/) 中，核心循环是：意图或提示 -> 工具调用 -> 观察结果 -> 更新计划 -> 最终输出。

本项目把这一模式映射到围串标领域的审查主循环中。

## 审查主循环

1. 定义审查任务。
2. 使用明确的解析后端加载招标文件和投标文件。
3. 把文本和元数据标准化为结构化证据。
4. 对每份文档抽取可疑信号。
5. 在过滤招标模板共用语句后，进行投标人两两比较。
6. 计算风险分并记录可解释的 finding。
7. 持久化整套运行产物，供人工复核和后续 agent 复用。

## 系统分层

### 1. 接口层

- CLI 命令
- 用于迭代审查的 REPL 模式
- 供下游自动化消费的 JSON 输出

### 2. 编排层

- 校验输入
- 创建运行目录
- 调用解析、抽取、评分与报告生成
- 写出清单与摘要

### 3. 输入与解析层

- `.txt`、`.md`、`.json` 的纯文本加载
- `.docx` 的 OOXML 文本抽取
- `.pdf` 优先使用 `pdftotext`，缺失时回退 `pypdf`
- 面向真实采购包的目录与 zip 递归加载

### 4. 能力层

- OCR 与扫描页文本抽取
- 图像理解与页面描述
- 表格抽取与标准化
- 文件元数据与签名提取
- 可服务多个业务流程的原子化输出

### 5. 领域分析层

- 采购文档中的主体信息抽取
- 报价信号抽取
- 罕见文本与重合片段检测
- 招标模板过滤

### 6. 证据与报告层

- 按文档标准化的 JSON
- 两两比较报告
- 人类可读的 Markdown 摘要
- 模板或 LLM 生成的审查意见书

## 设计原则

- 证据优先：每个分值都必须引用底层证据。
- 确定性基线：核心检查不依赖 LLM。
- 平滑的 LLM 增强：LLM 可用于意见草拟，但证据抽取与评分不依赖它。
- 产物对 agent 可读：输出必须便于下一次 agent 运行接续。
- 后端可扩展：文档解析和评分规则保持模块化隔离。
- 原子能力与业务判断分离，避免未来工具逻辑继续堆进 `core/`。
- 业务判断优先：先定义判断维度，再扩展事实层、策略层、OCR 或 LLM。

## 后续扩展方向

- OCR 与扫描 PDF 支持继续增强
- 更丰富的报价表和清单表比较
- 基于已抽取证据的 LLM 调查纪要
- 审查基准数据集与回归 harness
