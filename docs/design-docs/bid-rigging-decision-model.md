# 围串标业务判断层要素模型

## 目的

本文件用于定义围串标审查中的核心业务判断模型，聚焦回答一个问题：

**多家投标人之间，是否存在足以支持围串标怀疑的异常信号。**

本文件位于 `ReviewFacts`、`strategy`、OCR、LLM 之上。系统应先明确“要判断什么”，再反推“需要哪些事实”，最后才决定“调用哪些能力”。

## 设计顺序

本仓库后续应遵循以下设计顺序：

1. 业务判断模型
2. 统一事实模型
3. 审查策略
4. 原子能力
5. 报告输出

这意味着：

- `scoring` 应由清晰的业务判断维度驱动。
- `ReviewFacts` 只保留服务这些判断维度的事实。
- `strategy` 只在事实缺失或证据不足时请求 OCR、LLM、表格抽取等能力。

## 核心判断目标

业务层并不直接给出法律意义上的最终定性，而是对每一组投标人输出以下三类判断之一：

1. `no_clear_signal`
   当前材料未显示出足够明确的异常信号。
2. `needs_follow_up`
   已存在可疑线索，但当前证据不足，需要继续补充核查。
3. `priority_review`
   已存在多项强异常信号，应列为重点复核对象。

因此，系统输出的重点应是：

- 供应商两两之间的可疑程度判断
- 案件层面哪些供应商组合最值得重点关注
- 每个判断背后的证据解释

## 判断维度

围串标业务判断层建议固定为六个维度。

### 1. 主体关联维度

问题：
不同投标人之间，是否存在直接或间接的主体关联？

典型信号：

- 法定代表人相同
- 授权代表相同
- 联系电话相同
- 邮箱相同
- 银行账户相同
- 地址相同
- 统一社会信用代码相同

解释：

- 这是最强的一类直接关联维度
- 应当给予较高权重
- 应尽量降低误报

### 2. 报价关联维度

问题：
报价结果是否更像协同形成，而不是独立竞争形成？

典型信号：

- 投标总报价完全一致
- 投标总报价异常接近
- 分项报价结构异常相似
- 舍入规律一致
- 分项报价分布不符合独立竞争特征

解释：

- 这是重要维度，但通常不能单独定性
- 应与主体关联、文件同源、文本方案等其他维度联读

### 3. 文本与方案关联维度

问题：
不同投标文件之间，是否存在异常相似的文本或方案内容？

典型信号：

- 罕见措辞重合
- 共同错误
- 相同的自定义叙述结构
- 技术方案章节明显相似
- 实施方案或培训方案在去模板后仍高度相似

解释：

- 这一维度天然误报风险较高
- 必须强力降权以下内容：
  - 标准投标模板
  - 平台流程说明
  - 通用售后/培训承诺
  - 规范性响应章节

### 4. 结构同源维度

问题：
不同投标文件的结构是否显示出共同制作或复用痕迹？

典型信号：

- 章节顺序相同
- 缺失章节相同
- 文件/组件结构相似
- 文件命名风格相似
- 表格布局风格相似
- 可编辑文件指纹相似

解释：

- 这是重要的辅助维度
- 在多文件投标包场景下尤其重要
- 在长 PDF 场景下，应更重视章节结构相似，而不是文件树相似

### 5. 授权与资质链维度

问题：
不同投标人之间，是否共享异常重叠的授权或资质材料？

典型信号：

- 厂家授权链相同
- 注册证号相同
- 许可证号相同
- 授权时间重叠异常
- 资质材料疑似复用

解释：

- 当能拿到稳定标识时，这一维度价值较高
- 需要较好的来源定位能力

### 6. 时间与电子痕迹维度

问题：
电子痕迹是否显示出共同准备、同设备生成或协同投标特征？

典型信号：

- 创建/修改时间相同或高度集中
- CA 证书使用人相同
- 上传终端或 IP 相同
- 文档指纹相同
- 页级或图片生成特征相似

解释：

- 这是高价值维度
- 但现实中往往材料不全，因此系统要允许缺失，不应强行补推断

## 所需事实族

业务判断层反推出 `ReviewFacts` 至少应包含以下事实族。

### 主体事实

- 企业名称
- 法定代表人
- 授权代表
- 电话
- 邮箱
- 地址
- 银行账户
- 统一社会信用代码

### 报价事实

- 投标总报价
- 分项报价
- 报价来源章节
- 报价来源页码

### 文本与章节事实

- 章节标题
- 章节家族分类
- 罕见重合片段
- 重合来源文件和页码
- 共同错误片段

### 结构事实

- 章节目录
- 章节顺序
- 文件/组件清单
- 文件指纹

### 授权与资质事实

- 制造商
- 授权方
- 授权时间
- 注册证号
- 许可证号

### 时间与电子痕迹事实

- 创建/修改时间
- 上传时间
- 证书/操作人线索
- 哈希或文件指纹

## 事实质量规则

业务判断层消费事实时，应遵循以下原则：

1. 稳定标识优先于叙述相似。
2. 有来源定位的事实优先于纯推断事实。
3. 主采纳值优先于候选值。
4. 缺失应视为 `unknown`，而不是 `negative`。
5. 文本相似不能单独升级结论，除非该文本明确罕见且非规范性。

## 维度权重指导

这不是最终固定分值表，而是业务层优先级指导。

- 最高优先级：
  - 主体关联
  - 时间与电子痕迹
- 高优先级：
  - 报价关联
  - 授权与资质链
- 中优先级：
  - 结构同源
- 条件优先级：
  - 文本与方案关联
    仅在完成强模板过滤和规范性降权后才应提升权重

## 各维度证据强弱分层

为避免不同维度中的信号被同等看待，业务判断层应进一步区分“强证据 / 中证据 / 弱证据”。

统一原则如下：

- 强证据：单条即足以明显抬高案件优先级。
- 中证据：需要与其他维度的证据组合判断。
- 弱证据：只能作为辅助背景，不能单独推高结论。

### 1. 主体关联维度

强证据：

- 统一社会信用代码相同
- 银行账户相同
- 联系电话相同
- 邮箱相同
- 同一授权代表且身份证件信息能够对应

中证据：

- 法定代表人相同
- 授权代表姓名相同但缺少更强标识
- 地址高度一致且不是园区/代理地址
- 联系人姓名相同，且伴随其他辅助字段相似

弱证据：

- 地址位于同一楼宇或同一园区
- 法定代表人同名但不能确认是同一人
- 普通联系人姓名相同
- 注册地位于同一区县

### 2. 报价关联维度

强证据：

- 投标总报价完全一致，且项目不属于明显标准化限价场景
- 投标总报价一致，同时多个分项报价也同步一致
- 分项报价结构、税率、尾数规律高度一致

中证据：

- 投标总报价极度接近
- 多个分项报价接近且结构高度相似
- 报价拆分方式相似，且差额分布不自然

弱证据：

- 总报价处于同一区间
- 只有总价接近，没有分项支撑
- 低价/高价排序看起来相似

### 3. 文本与方案关联维度

强证据：

- 罕见且个性化表述重合
- 多处共同错误
- 专业方案中出现相同的异常逻辑、异常术语或异常排版错误
- 仅两家共享，且明显非模板、非规范响应的核心段落重合

中证据：

- 技术方案、实施方案、培训方案存在较多相似片段
- 章节叙述结构明显相似
- 相同的自定义表述，但尚未达到“罕见错误”级别

弱证据：

- 售后服务承诺相似
- 培训方案通用话术相似
- 平台功能说明相似
- 投标函、承诺书、偏离表、业绩表等规范响应内容相似

### 4. 结构同源维度

强证据：

- 文件指纹一致
- 可编辑源文件明显同源
- 文件结构、缺失项、页序、表格布局同时高度一致，且伴随其他异常

中证据：

- 章节顺序高度一致
- 多份文档结构同构
- 同样缺失某些不常见章节
- 组件命名和装配方式高度相似

弱证据：

- 目录结构相似
- 表格格式接近
- 文件命名风格类似
- 都按招标文件顺序响应

### 5. 授权与资质链维度

强证据：

- 同一注册证号、许可证号被异常共用
- 同一厂家授权链在竞争供应商间高度重叠且时间异常一致
- 授权材料存在明显同源复用痕迹

中证据：

- 授权厂家相同
- 授权时间接近
- 资质材料内容相似
- 多项资质编号、覆盖产品、时间范围存在异常重合

弱证据：

- 都使用同品牌、同厂家体系
- 都具备同类别资质
- 授权格式相似但编号不同

### 6. 时间与电子痕迹维度

强证据：

- 同一上传 IP 或终端
- 同一 CA 证书使用人
- 同一设备或同源文件指纹
- 创建/修改时间高度集中，且与文件生成逻辑一起支持同源判断

中证据：

- 多个文件生成时间接近
- PDF/Office 元数据显示相同作者、软件或环境
- 多家文件组件时间轨迹呈明显同步特征

弱证据：

- 时间接近但样本过少
- 只有软件版本相同
- 只有普通元数据相似，没有更强电子痕迹支撑

## 初步结论升级规则

在第一阶段的业务判断中，可先使用以下简单规则：

- 出现 1 条强证据：至少进入 `needs_follow_up`
- 出现 2 条及以上中证据，且来自不同维度：进入 `needs_follow_up`
- 出现 1 条强证据 + 1 条中证据，或 2 条强证据：进入 `priority_review`
- 只有弱证据：维持 `no_clear_signal`

该规则仅作为第一阶段的业务判断基线，后续可再细化成更正式的评分矩阵。

## 评分输出契约

`scoring` 对每一组供应商至少应输出：

- `judgment`
  - `no_clear_signal`
  - `needs_follow_up`
  - `priority_review`
- `risk_level`
  - low / medium / high / critical
- `dimension_summary`
  各判断维度的贡献摘要
- `findings`
  与具体维度绑定的证据项

这样 `scoring` 就不是简单分数累加，而是一个按维度组织的业务判断引擎。

## 对 ReviewFacts 的约束

`ReviewFacts` 不应成为一个泛化的大字典，而应是服务上述六个判断维度的统一事实层，且每个事实至少应包含：

- 主采纳值
- 候选值
- 来源文件
- 来源页码
- 来源类型
- 置信度
- 必要时的冲突标记

如果某一类数据不能支撑六个判断维度之一，就不应提升为核心事实。

## ReviewFacts 主采纳值与候选值分层建议

为避免 `ReviewFacts` 失控膨胀，应将事实分为三层：

1. 主采纳值  
   直接参与业务判断、可稳定归一化、适合做供应商间比对的字段。
2. 候选值  
   有一定价值，但稳定性、歧义性或质量还不足以直接作为主判断输入的字段。
3. 辅助证据  
   更适合用于解释、溯源、人工复核，不适合作为单值主字段的内容。

总原则：

- 能稳定归一化、能直接参与 pairwise 比对的，优先进入主采纳值。
- 歧义较大、误识别风险较高、或本质上是比较结果而不是原始事实的，保留为候选值或辅助证据。
- 比较型结果（如“文本相似”“结构相似”）不应硬塞为供应商单体主字段。

### 一、建议进入主采纳值的字段

这些字段应优先作为 `ReviewFacts` 的核心单值事实：

#### 主体关联维度

- 企业名称
- 法定代表人
- 联系电话
- 邮箱
- 银行账户
- 统一社会信用代码

#### 报价关联维度

- 投标总报价

#### 授权与资质链维度

- 制造商
- 授权厂家
- 注册证号
- 许可证号

#### 可选主采纳值

- 地址

地址可保留主采纳值，但业务上应默认视作弱字段，并保留多个候选来源，以避免园区地址、代理注册地址、断句污染带来的误判。

### 二、建议保留为候选值的字段

这些字段可能有审查价值，但稳定性不足，不宜直接作为主判断输入：

#### 主体关联维度

- 授权代表
- 联系人姓名
- 地址的其他候选值
- 单位性质
- 成立时间

#### 报价关联维度

- 分项报价
- 税率
- 含税/不含税拆分
- 报价备注
- 分润比例、特殊计价说明

#### 授权与资质链维度

- 授权时间
- 授权覆盖范围
- 资质材料标题
- 证照图片提取结果

#### 时间与电子痕迹维度

- 创建时间
- 修改时间
- 上传时间
- CA 使用人
- 组件级时间点

### 三、建议保留为辅助证据的内容

这些内容更适合作为解释材料、附表、引用片段或人工复核输入，而不是主事实字段：

#### 文本与方案关联维度

- 技术方案章节正文
- 实施方案章节正文
- 培训方案章节正文
- 罕见重合片段
- 共同错误片段
- 章节标题与页码

#### 结构同源维度

- 章节目录
- 章节顺序
- 文件组件列表
- 文件指纹
- 表格结构摘要

#### 时间与电子痕迹维度

- 页级生成痕迹
- 文件元数据原文
- 终端/IP 原始日志

### 四、不应直接进入主采纳值的内容

以下内容虽然重要，但本质上属于“比较结果”，不应被当作供应商单体事实保存为主值：

- 文本相似度结果
- 结构相似度结果
- 章节相似度结果
- 风险评分结果
- 异常等级结果

这些应由 `scoring` 或证据表生成，而不是进入 `ReviewFacts` 的核心字段。

### 五、对 fusion 的直接约束

`fusion` 在构建 `ReviewFacts` 时，应遵循以下规则：

1. 先尝试产出主采纳值。
2. 如果主值不稳，则保留候选值，不强行主采纳。
3. 如果是叙述性内容、章节内容、比较性结果，则进入辅助证据层。
4. 在主值和候选值之间保留来源页、来源文档、来源类型和置信度。

### 六、对 strategy 的直接约束

`strategy` 不应为了“多抽一点”而调用能力，而应围绕主采纳值缺口来补事实：

- 如果缺报价主值，则优先补报价表、开标一览表、报价单。
- 如果缺主体主值，则优先补营业执照、基本情况表、法代证明、授权书。
- 如果缺资质主值，则优先补注册证、许可证、授权材料。
- 如果主采纳值已经齐全，则不应仅因辅助证据不足而盲目扩大 OCR/LLM 范围。

### 七、当前实施优先级

对于当前仓库，第一阶段最值得稳定进入 `ReviewFacts` 主采纳值的字段是：

- 企业名称
- 法定代表人
- 联系电话
- 邮箱
- 银行账户
- 统一社会信用代码
- 投标总报价
- 制造商
- 授权厂家
- 注册证号
- 许可证号

地址作为保留字段可以存在，但在业务判断层默认按弱字段处理。

- 授权时间
- 授权覆盖范围

#### 时间与电子痕迹维度

- 创建时间
- 上传时间
- 证书 / 操作人线索
- 更稳定的文件哈希与电子痕迹承载

### 第一批优先落地字段

为尽快服务业务判断层，第一批优先落地字段建议为：

1. 统一社会信用代码
2. 授权代表
3. 分项报价结构
4. 授权厂家 / 授权方
5. 授权时间
6. 文件指纹

其中第一阶段最先进入实现的是：

- 统一社会信用代码
- 授权代表

原因：

- 这两个字段都直接服务主体关联维度
- 都可以较自然地进入 `ReviewFacts`
- 都能为后续 `scoring` 和 `strategy` 提供更稳的高价值事实入口

## 对 strategy 的约束

`strategy` 应该由“维度缺口”驱动，而不是由固定文件名或固定能力配置驱动。

例如：

- 如果报价事实薄弱，则触发报价表抽取或报价页 OCR。
- 如果主体事实薄弱，则优先补营业执照、基本情况表、法代证明、授权章节。
- 如果当前只有文本相似线索，而没有主体、报价、时间等支撑，则应避免过度升级。
- 如果已经出现强主体关联或强时间痕迹，则可以升级到更深的复核和更完整的报告。

所以 `strategy` 应回答：

- 哪个判断维度当前证据不足
- 哪种能力最适合补这个缺口
- 该能力的调用成本是否值得

## LLM 参与边界与触发条件

LLM 在围串标审查中应被定义为**辅助判断层**，而不是第一判断层，更不是最终唯一裁判。

基本原则：

- 业务判断层优先，LLM 后置介入。
- 规则和事实层先给出初步判断。
- 只有当规则判断不稳、证据存在解释空间或需要形成更完整审查表达时，才让 LLM 参与。

### 一、LLM 适合参与的场景

#### 1. 边界案件解释

当当前证据不足以直接进入明确结论，但存在一定可疑线索时，LLM 适合参与：

- 只有文本相似，但是否属于模板/规范文本尚不稳定
- 报价异常，但主体关联不足
- 结构相似存在，但无法单独判断其异常性

此时 LLM 的任务是：

- 解释为什么当前线索可疑
- 解释为什么当前仍不足以定性
- 给出更审慎的审查口径

#### 2. 多维证据冲突

当不同维度的证据方向不一致时，LLM 适合参与综合归纳。

例如：

- 报价关联明显，但主体关联缺失
- 文本与方案相似，但时间与电子痕迹没有支持
- 主体字段部分重合，但资质链没有异常

此时 LLM 的任务是：

- 对冲突证据进行综合解释
- 避免报告中出现前后矛盾
- 帮助形成“需继续核查”的稳定表达

#### 3. 事实缺口提示

当案件进入 `needs_follow_up`，但还不够进入 `priority_review` 时，LLM 适合总结当前证据缺口。

此时 LLM 的任务是：

- 当前最缺哪些关键事实
- 建议优先核查哪些材料或外围证据
- 哪些线索应继续延伸取证

#### 4. 审查表达与报告增强

当事实层和评分层已较稳定时，LLM 适合参与：

- 正式报告润色
- 审查意见归纳
- 风险摘要组织
- 可疑点与排除性因素的平衡表述

### 二、LLM 不应替代的部分

以下内容不应交给 LLM 直接裁定：

- 原始事实抽取结果的唯一来源
- 主体关联的最终机器判定
- 报价是否完全一致的基础计算
- 文件指纹、时间元数据等客观值
- 是否构成围串标的最终法律定性

也就是说，LLM 不能跳过事实层和规则层直接给出结论。

### 三、LLM 触发条件

建议 `strategy` 在以下情况下触发 LLM：

#### 1. 用户明确要求

- 明确要求“大模型审查”
- 明确要求正式增强版报告
- 明确要求对可疑点进行自然语言解释

#### 2. 规则判断处于边界区

- 当前判断为 `needs_follow_up`
- 证据主要集中在中证据和弱证据
- 文本与方案维度对结论影响较大

#### 3. 存在跨维度冲突

- 两个及以上维度方向不一致
- 报告中存在“异常点与排除性因素并存”的复杂情况

#### 4. 需要形成正式对外表达

- 要生成更正式的审查报告
- 要生成更像人工审查员撰写的意见书
- 要输出“事实 + 解释 + 建议”的完整结果

### 四、LLM 的输出定位

LLM 输出应定位为以下三类之一：

1. 证据解释  
   解释某条线索为什么可疑、为什么不足、为什么应降权。

2. 综合判断说明  
   对多个维度的证据进行归纳，形成清晰的审查表达。

3. 后续核查建议  
   指出下一步最值得优先核查的外围证据或文件。

### 五、LLM 的输入要求

LLM 不应直接吃未经整理的原始全文，而应优先基于以下结构化输入：

- `ReviewFacts` 主采纳值
- 候选值与冲突摘要
- 维度级证据摘要
- 风险评分结果
- 排除性因素摘要
- 关键文本片段和来源位置

这样可以保证：

- LLM 输出可控
- 解释与业务判断层一致
- 不会因上下文噪声过大而偏移判断

### 六、当前架构约束

对当前仓库而言，应固化以下规则：

- `scoring` 先给出初步业务判断
- `ReviewFacts` 先统一事实
- `strategy` 再判断是否值得触发 LLM
- `llm_review` 负责解释、润色、补充判断，不负责替代规则结论

一句话总结：

**LLM 应作为业务判断层的后置辅助层，在规则判断不稳、证据有解释空间、或需要更完整审查表达时介入，而不应越过事实层和规则层直接判案。**

## 对长 PDF 场景的含义

在长 PDF 场景下，这个模型意味着：

- 章节事实必须是一等公民
- 报价抽取必须基于章节定位
- 技术方案/实施方案/培训方案比较必须按章节进行
- OCR 应作为扫描页补漏手段，而不是默认主入口

## 六个维度做实的执行清单

本节用于把“业务判断层优先”的原则继续落到项目执行上。目标不是继续增加维度数量，而是先把现有六个维度做实，确保：

- 每个维度都有稳定的事实输入
- 每个维度都能形成明确的强 / 中 / 弱判断
- 每个维度都能进入报告表达
- `strategy` 的触发逻辑能够围绕维度缺口展开

### 一、主体关联维度

#### 目标

稳定识别并比较不同投标人之间是否存在直接或近直接主体关联。

#### 当前已有

- 企业名称
- 法定代表人
- 授权代表
- 联系电话
- 邮箱
- 银行账户
- 地址
- 统一社会信用代码

#### 仍需补强

- 联系人姓名单列
- 地址标准化与去噪
- 同名人员去歧义
- 主采纳值与候选值冲突处理
- OCR 场景下主体字段保底抽取

#### 执行动作

1. 增加 `contact_names` 等主体辅助字段。
2. 对地址做标准化和弱证据降权。
3. 为法定代表人、授权代表增加来源和置信度摘要。
4. 让主体关联维度直接输出强 / 中 / 弱结论，而不是只散落在 `findings` 中。

### 二、报价关联维度

#### 目标

不只识别总报价，还要识别报价结构是否存在异常一致或异常接近。

#### 当前已有

- 投标总报价
- 分项报价结构 `pricing_rows`

#### 仍需补强

- 分项报价行归一化
- 税率、含税 / 不含税拆分
- 特殊计价说明
- “极度接近”与“结构一致”阈值
- 长 PDF 场景下报价表稳定抽取

#### 执行动作

1. 统一 `pricing_rows` 的字段结构。
2. 增加税率、备注、报价说明等辅助字段。
3. 明确总报价和分项报价的判断阈值。
4. 在 `scoring` 中单独表达“总价异常”和“结构异常”。

### 三、文本与方案关联维度

#### 目标

只保留真正有判断价值的文本相似内容，把模板、规范响应、行业通用语句全部降噪。

#### 当前已有

- 文本相似规则
- 常见模板 / 售后 / 培训 / 偏离表 / 承诺书等大范围降权
- 文本重合证据附表
- 页码 / 行号 / 来源段落定位

#### 仍需补强

- 罕见表达识别
- 共同错误识别
- 章节类型标签更细
- 长 PDF 场景下技术 / 实施 / 培训方案差异化判断
- “个性化片段”与“规范响应片段”分类

#### 执行动作

1. 增加“共同错误”与“罕见表达”规则。
2. 为文本片段增加 `template / normative / personalized` 标签。
3. 让正式报告只展示高价值片段。
4. 把一般相似片段继续从重点附表中降出。

### 四、结构同源维度

#### 目标

识别是否存在同源制作、同底稿加工、异常同构结构。

#### 当前已有

- 文档分类
- 章节切分
- 章节级比对
- 一部分文件哈希 / 组件信息
- `section_similarity_table`

#### 仍需补强

- 文件指纹正式进入 `ReviewFacts`
- 章节顺序摘要
- 表格结构摘要
- 同样缺失章节的规则
- 长 PDF 与多文件包统一结构画像

#### 执行动作

1. 补 `file_fingerprints`、`section_order_profile` 等结构字段。
2. 将结构同源从“相似描述”提升为可评分维度。
3. 增加“异常同构”规则，而不只是一般相似。
4. 在报告中单列结构同源维度判断。

### 五、授权与资质链维度

#### 目标

识别授权链、厂家链、资质链是否存在异常重合、异常复用或异常同步。

#### 当前已有

- 制造商
- 授权厂家
- 授权方
- 授权时间
- 注册证号
- 许可证号

#### 仍需补强

- 授权对象
- 授权覆盖范围
- 资质页来源摘要
- OCR 对证照页的稳定抽取
- “正常同品牌”与“异常授权重叠”的解释规则

#### 执行动作

1. 增加 `authorization_targets`、`authorization_scopes`。
2. 强化注册证、许可证、授权书的来源页绑定。
3. 让 `scoring` 区分“厂家相同”与“授权链异常重叠”。
4. 在报告里使用更审慎的授权链表达。

### 六、时间与电子痕迹维度

#### 目标

逐步把最硬的一类同源证据纳入统一事实层和评分层。

#### 当前已有

- 部分修改时间
- 时间摘要

#### 仍需补强

- 创建时间
- 文件哈希 / 指纹主字段
- 上传时间
- CA 使用人
- 终端 / IP
- 页级 / 组件级生成轨迹
- 更可靠的时间线规则

#### 执行动作

1. 先补创建时间、修改时间、文件指纹。
2. 再补组件级时间轨迹。
3. 为未来平台日志和电子痕迹预留字段。
4. 明确“有则高权重，无则不扣分”的业务原则。

### 七、跨维度共性执行项

#### 1. 做实 `ReviewFacts`

- 主采纳值 / 候选值 / 辅助证据继续分层
- 强化来源页码、来源文档、置信度
- 增加冲突标记与主采纳理由

#### 2. 做实 `scoring`

- 每条 `finding` 必须归属明确维度
- 每个维度必须形成 `strong / medium / weak / none`
- 结论逐步从“总分导向”升级为“维度组合导向”

#### 3. 做实报告表达

- 正文按维度解释，不只给总分
- 只展示高价值证据片段
- 模板性内容继续降噪
- 结论、维度摘要、附表要保持一致

#### 4. 做实 `strategy`

- OCR / 表格 / LLM 的触发要围绕“哪个维度缺事实”展开
- 不为了启用能力而启用能力
- 优先解决维度缺口，再决定是否进入更深审查

### 八、优先级顺序

#### 第一优先级

- 主体关联
- 报价关联
- 授权与资质链

#### 第二优先级

- 结构同源
- 文本与方案关联

#### 第三优先级

- 时间与电子痕迹

### 九、建议推进顺序

1. 继续补稳主体关联和报价关联。
2. 把授权与资质链规则做细。
3. 把结构同源真正接进 `ReviewFacts` 和 `scoring`。
4. 把文本维度升级成“共同错误 / 罕见表达优先”。
5. 最后深挖时间与电子痕迹。

## 非目标

本模型不负责：

- 给出最终法律定性
- 代替人工复核
- 假设所有异常信号具有相同意义
- 要求每个案件都具备全部六个维度的完整证据

## 当前结论

对当前仓库的后续约束可以总结为一句话：

**先定义业务判断维度，再定义事实层；先有事实层，再决定策略层；最后再调度 OCR、表格、LLM 等能力。**
